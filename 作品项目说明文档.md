# Terminal Copilot（终端场景的 AI 知识执行引擎）— 作品项目说明文档

这份文档用于说明本作品解决的问题、核心价值、AI 能力点、技术实现与交互设计。

GitHub 仓库：https://github.com/panbu2007/Terminal-Copilot

---

## 0. 一句话结论

Terminal Copilot 不是另一个只会生成命令的终端 AI，而是一套面向终端场景的 AI 知识执行引擎：把团队操作经验变成可检索的依据、可审查的执行计划、可导出的审计结果，解决企业把 AI 引入终端时最核心的信任问题。

---

## 1. 项目简介

### 1.1 背景与痛点

现在几乎所有终端 AI 工具都面临同一个信任危机：

- AI 给了一条命令，我敢直接执行吗？
- AI 在生产环境里做了修改，谁负责？事后怎么追溯？
- 团队经验都在老员工脑子里，AI 为什么能理解我们的 SOP？
- 给新人一个 AI 工具，不等于教会他完整流程。

这正对应终端场景里的四个真实痛点：

- **AI 命令信任危机**：用户无法判断 AI 建议是否可靠，只能自己背锅。
- **AI 操作审计黑洞**：执行结束后缺少结构化日志、责任链和复盘材料。
- **团队知识黑箱**：组织内部 Runbook 无法稳定进入 AI 的决策过程。
- **新人培训瓶颈**：零散命令建议无法替代标准化 SOP 和逐步引导。

### 1.2 目标用户

- 运维/平台/DevOps 团队：需要让 AI 进入终端流程，但必须保留审查、风控和审计能力
- 企业技术团队：希望把 Runbook、SOP、历史经验转成可复用的智能工作流
- 新同学/学生/运维新人：需要沿着标准流程完成排错，而不是自己猜下一步
- 评审/答辩场景：需要一个可演示、可解释、可追溯的 AI 系统原型

### 1.3 产品定位

**Terminal Copilot：终端场景的 AI 知识执行引擎**

把团队操作经验变成可检索、可执行、可审计的智能工作流。

它不是单纯的“命令生成器”，而是把终端 AI 的完整可信闭环做出来：

- **知识资产化**：Runbook -> 检索 -> AI 引用依据
- **执行前可视化审查**：先看 DAG 计划、风险、分支、审批点，再决定是否执行
- **执行后自动审计**：生成结构化报告，支持回放、导出和复盘

## 2. 核心价值与差异化

### 2.1 为什么能打中行业真痛点

行业真正缺的不是“更会写命令的 AI”，而是“能为 AI 命令负责的机制”。

Terminal Copilot 把“能用”推进到“敢用、可用、可管”：

- **敢用**：执行前先审查完整路径，不是直接让用户盲点执行
- **可用**：结合内部 Runbook 和上下文，建议更贴近真实团队流程
- **可管**：执行后自动产出审计证据链，便于责任追溯和合规留痕

### 2.2 与 Claude Code / Warp 的差异

Claude Code、Warp 等工具的强项在于命令生成与交互效率，但在企业落地时仍然留下三类空缺：

- **缺少执行前审查层**：通常只展示命令文本，不展示完整执行路径、依赖关系和审批点
- **缺少执行后审计层**：执行完后缺少结构化 PASS/WARN/FAIL 报告，难以追溯
- **缺少组织知识层**：不了解团队内部 Runbook、标准 SOP 和环境约束

Terminal Copilot 的差异化根基正是围绕这三层建立：

- 让 AI 命令在执行前先接受可视化审查
- 让执行过程和结果自动沉淀为审计资产
- 让组织知识进入 AI 决策链路，而不是停留在文档里

### 2.3 Web 形态为什么是优势

Web 不是弱点，而是这个产品成立的前提。

因为以下能力天然更适合 Web：

- DAG 执行计划图与交互式审查
- 团队共享与评审协作
- 审计面板、导出和留痕
- 权限控制与审批体验

CLI 适合“个人即时操作”，Web 更适合“团队级可信执行”。

## 3. 三大核心能力

### 3.1 知识资产化

- 组织 Runbook 进入知识库，形成运行时可检索依据
- 建议可以附带来源 `source` 和摘要 `snippet`
- 支持上传自定义 Markdown runbook，降低组织知识接入成本

在技术上：

- `docs/runbook/*.md` 是当前运行时知识源
- `rag.py` 提供 BM25 + 标题/关键词增强检索
- `rag_v2.py` 提供 embedding 驱动的混合向量检索 + RRF 融合
- embedding 默认可使用 Modelscope 的 `BAAI/bge-small-zh-v1.5`，也支持切换 provider/model

### 3.2 可视化执行计划

- AI 不是直接给一条命令，而是生成可审查的执行计划
- 计划以 DAG 形式展示步骤依赖、条件分支、风险和验证点
- 用户可以在执行前看到完整路径，在执行中实时追踪节点状态

在交互上：

- 支持审批节点
- 支持跳过/取消
- 支持实时 stdout/stderr
- 支持计划前 pre-audit 风险提示

### 3.3 自动审计

- 执行结束后自动汇总为结构化审计报告
- 对每个节点标记状态、风险、是否 grounded、输出摘要
- Safety Agent 会补充总体分析、发现项和建议动作

最终输出可作为：

- 团队复盘材料
- 交接材料
- 合规审计留痕
- 比赛/答辩中的可视化证据链

## 4. 业务价值说明

### 4.1 对用户的价值

- **降低门槛**：把经验型终端变成流程型终端
- **减少试错**：先看计划、再执行、再验证，降低误操作概率
- **提升可解释性**：建议带依据，不再是黑盒输出
- **形成可回放过程**：做完一件事后，过程不会消失

### 4.2 对组织的价值

- **SOP 资产化**：把隐性经验转成 AI 可调用的知识层
- **流程标准化**：让新人沿着统一 DAG 路径操作
- **风险可控**：通过 pre-audit、policy、审批降低高危执行风险
- **审计可追溯**：把终端操作变成可导出、可复盘的证据链

### 4.3 可量化指标

- 一次典型故障排查从“搜索 + 试错”缩短为“检索 + 审查 + 执行”
- 新人完成标准 SOP 的成功率提高
- 高风险命令的误执行概率下降
- 团队内部重复答疑与口头交接成本下降

## 5. AI 创新性说明

### 5.1 AI 在系统中的位置

- **Planner**：根据用户意图、历史上下文和规则生成建议与执行计划
- **RAG**：从组织知识中检索可引用依据
- **Safety Agent**：对建议和计划做风险分析与审计总结
- **Verifier**：把“验证成功”转成可执行检查步骤

### 5.2 为什么 AI 是核心

- **核心不是补全命令，而是补全工作流**
- **核心不是聊天，而是执行前审查 + 执行后审计**
- **核心不是黑盒判断，而是把依据、路径、结果显式化**

### 5.3 创新点

- **把终端 AI 的信任机制产品化**：不是只生成命令，而是生成可信执行闭环
- **知识、执行、审计三层闭环**：不是单点功能，而是完整系统能力
- **轻量多 Agent 协作**：diag/rag/executor/safety 分工明确，提升可解释性
- **混合 RAG + 可视化 DAG**：让“为什么这么做”和“接下来怎么做”同时可见

## 6. 技术实现说明

### 6.1 整体架构

- 前端：纯静态页面 + Web 终端 + 计划图 + 审计面板
- 后端：FastAPI 统一托管 API 与静态资源
- 执行器：
  - `simulate`：安全演示模式
  - `local`：本地执行模式，支持 cwd、内建 `cd`、中断
- 检索：
  - `rag.py`：关键词/BM25 检索
  - `rag_v2.py`：混合向量检索 + RRF 融合
- Agent：
  - orchestrator 协调 diag/rag/executor/safety

### 6.2 核心数据流

1. 用户输入问题、意图或报错
2. Planner 生成建议或 DAG 执行计划
3. RAG 补充引用依据
4. Safety Agent 对建议/计划做 pre-audit
5. 用户批准后进入执行
6. 执行过程通过 SSE 流式推送节点状态与输出
7. 结束后自动生成 execution audit 报告并支持导出

### 6.3 稳定性与可部署性

- 后端直接托管前端，不依赖 Node.js 构建链
- 未配置 LLM/embedding 时仍可回退到规则 + 关键词检索
- 执行器模式可切换，方便真实部署和安全演示
- 计划、会话、审计均以服务端内存状态驱动，适合原型与比赛演示

## 7. 交互与设计说明

### 7.1 核心界面

- 左侧：真实 Web 终端
- 右侧：
  - 建议区
  - DAG 执行计划图
  - Agent 协作面板
  - Audit 审计面板
  - 时间线与导出能力

### 7.2 关键流程

1. **输入问题或失败上下文**
2. **AI 生成带依据的建议或执行计划**
3. **用户先审查，再批准执行**
4. **系统实时展示执行进度**
5. **执行后自动输出审计报告和下一步建议**

### 7.3 安全与可控体验

- 高危命令先经过 `policy.py`
- 执行计划可先做 pre-audit
- 审批节点避免高风险步骤被直接放行
- 审计面板保留执行证据链
