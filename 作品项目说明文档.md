# Terminal Copilot（Web 终端智能副驾）— 作品项目说明文档

这份文档用于说明本作品解决的问题、核心价值、AI 能力点、技术实现与交互设计。

GitHub 仓库：https://github.com/panbu2007/Terminal-Copilot

---

## 0. 一句话结论

Terminal Copilot 把“命令行靠记忆 + 反复试错”的工作方式，升级为“意图驱动 + 可执行建议 + 可验证闭环”的任务流：用户输入一句话或上下文，系统给出下一步命令（含 why/risk/rollback/verify），可一键执行并自动把过程沉淀到右侧时间线。

---

## 1. 项目简介

### 1.1 背景与痛点

在真实开发/运维/学习场景中，命令行的主要困难不在于“不会打字”，而在于：

- 不知道下一步该做什么（排错链路不完整、缺少 SOP）
- 害怕误操作（删库、停服务、覆盖配置等风险）
- 依赖碎片化经验（靠搜索/问人/复制粘贴）
- 任务过程不可回放（做完就忘，难复盘、难交接）

### 1.2 目标用户

- 新同学/学生/运维新人：不熟悉终端套路、需要“靠谱的下一步”
- 开发者：常见故障排查与固定流程希望更快、更稳
- 评审场景：需要可演示、可解释、可复现的 AI 工具产品

### 1.3 解决方案

Terminal Copilot 在 Web 终端里提供：

- **结构化建议**：命令 + 解释（why）+ 风险（risk）+ 回滚（rollback）+ 验证（verify）
- **可执行闭环**：一键执行建议命令 + 自动验证 + 下一步建议
- **可解释依据（RAG）**：建议可以引用本仓库的 runbook 文档片段（带 source）
- **安全护栏**：危险命令提示与二次确认
- **任务流时间线**：每次执行形成“回合”，可导出回放

## 2. 业务价值说明

### 2.1 对用户的价值

- **降低门槛**：把“经验型终端”变成“流程型终端”，新手也能走完排错闭环
- **减少试错**：把常见故障的诊断链路显式化（先只读→再修改→再验证）
- **减少复制粘贴**：建议可插入/一键执行，避免在网页/文档之间跳来跳去
- **可交接可复盘**：时间线 + 导出回放，让任务变成可复现资产

### 2.2 对组织/业务的价值

- **SOP 可资产化**：runbook + RAG 让隐性经验可检索、可复用
- **效率提升**：缩短排错时间，减少“人肉问答”与重复沟通
- **风险可控**：安全策略与确认流程减少高危误操作概率

### 2.3 可量化指标

- 完成一次典型排错闭环耗时（例如 60–90 秒内跑完）
- 复制粘贴次数（从“搜索→复制→试错”变为“一键执行/插入”）
- 失败重试次数（有 verifier/提示后明显减少）

## 3. AI 创新性说明

### 3.1 AI 在系统中的位置

- **Planner（建议生成）**：根据用户输入 + 上一次执行结果（stdout/stderr/exit_code）生成下一步建议
- **RAG（依据检索）**：从仓库内 runbook 文档检索相关片段，作为建议的“可追溯依据”
- **Policy（安全护栏）**：对危险命令进行 block/warn/confirm
- **Verifier（结果校验）**：将“执行后如何验证成功”变成具体命令步骤

### 3.2 为什么 AI 是核心（不是普通脚本/命令补全）

- **核心不是补全命令，而是补全任务链路**：把“下一步做什么”变成 AI 的决策输出
- **核心不是聊天，而是可执行闭环**：建议→执行→验证→下一步的循环才是产品价值
- **核心不是玄学，而是可解释**：RAG 引用 runbook 片段 + source，让建议可追溯

### 3.3 创新点

- **Infinite UI 思路**：右侧按回合聚合 Plan/Execute/Verify/Next，形成可滚动任务流
- **“宁缺毋滥”的引用策略**：检索阈值随 query 长度自适应，减少跑偏引用
- **轻量多 Agent 分工**：Planner/Policy/Verifier 在 UI 与导出事件里可被标注

## 4. 技术实现说明

### 4.1 整体架构

- 前端：纯静态页面（xterm.js）+ 任务流 UI（时间线/建议卡/确认卡）
- 后端：FastAPI 统一托管 API 与静态资源
- 执行器：
  - `simulate`：安全演示模式（不真正执行系统命令）
  - `local`：本地执行模式（带 cwd 与内建 cd，支持中断）
- 检索：对仓库内 `docs/runbook` 等 Markdown 做轻量检索与 rerank
- LLM：ModelScope（OpenAI-compatible chat completions），未配置 Token 时自动回退规则建议

### 4.2 核心数据流

1) 用户输入（意图/命令/报错）
2) 后端执行（或模拟执行）得到 stdout/stderr/exit_code
3) Planner 生成建议（规则优先，LLM 可选增强）
4) RAG 为建议补充引用依据（source + snippet）
5) Policy/Verifier 标注风险与验证步骤
6) 前端展示为建议卡 + 时间线回合，并支持一键执行

### 4.3 稳定性与可部署性

- 后端直接托管前端，降低部署门槛（不依赖 Node.js）
- Token 支持环境变量或本地落盘（`.secrets/`），并在界面中提示未配置状态
- 执行器可切换，便于比赛环境演示与安全展示

## 5. 交互与设计说明（关键流程/界面）

### 5.1 核心界面

- 左侧：Web 终端（支持提示符、历史↑↓、光标移动、Delete、Ctrl+C 中断等）
- 右侧：
  - 建议区：结构化建议卡（标题/命令/解释/风险/依据/按钮）
  - 任务流时间线：按回合展示 Plan / Execute / Verify / Next

### 5.2 关键流程

1) **输入问题或执行失败命令**（例如端口占用、缺依赖）
2) **得到“下一步建议”并一键执行**（必要时二次确认）
3) **自动给出验证步骤与下一步**，并在时间线形成可回放回合

### 5.3 安全与可控体验

- 高危命令：block 或 warn + 二次确认卡
- 错误/状态类建议：不提供“执行”按钮，避免误导
