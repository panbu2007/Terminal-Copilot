# 防火墙、安全组与端口连通性检查

适用场景：
- 服务本机正常，但外部无法访问
- 新服务上线后端口不通
- 需要快速判断问题在进程、主机防火墙、云安全组还是反向代理

症状：
- 浏览器或客户端连接超时
- `curl 127.0.0.1:PORT` 正常，但远端访问失败
- 服务端口监听正常，公网仍不可达

快速判断：

本机检查：

```bash
ss -ltnp | grep PORT
curl -v http://127.0.0.1:PORT
```

远端检查：

```bash
curl -v http://SERVER_IP:PORT
nc -vz SERVER_IP PORT
```

修复步骤：

1. 先确认进程真的在监听

```bash
ss -ltnp | grep PORT
```

- 如果只监听 `127.0.0.1`，外部访问必然失败
- 需要对外提供服务时，应用通常应监听 `0.0.0.0`

2. 检查本机防火墙

Linux 常见：

```bash
iptables -S
firewall-cmd --list-ports
ufw status
```

3. 检查云安全组 / 网络 ACL
- 确认入口规则放通目标端口
- 确认来源 IP 范围正确
- 确认出方向规则没有挡住回包或依赖调用

4. 检查反向代理和 LB
- 如果用户应访问 80/443，而不是业务原始端口：
  - 先检查 Nginx / LB 配置
  - 再检查 upstream 服务

5. 分层定位
- 本机 `127.0.0.1:PORT` 不通：进程问题
- 本机通、同网段不通：防火墙或监听地址问题
- 内网通、公网不通：安全组、LB、边界网络问题

回滚：
- 恢复上一个稳定防火墙规则
- 恢复上一个安全组模板
- 如果刚改了监听地址或代理配置，恢复旧配置并重启服务

验证：
- 本机访问通过
- 同网段访问通过
- 远端访问通过
- 反向代理或域名访问通过

风险提示：
- 不要为排障直接“全开放所有端口”
- 先确认访问路径再开规则，避免误暴露内部服务
- AI 面板、推理服务、数据库端口通常不应直接对公网开放

关键词：
- firewall port check
- security group port
- iptables port
- ufw port
- service listen 0.0.0.0
- 端口不通
- 安全组排查
- 防火墙检查
