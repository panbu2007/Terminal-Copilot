# 事故初始响应检查清单

适用场景：
- 收到生产告警，但根因还不明确
- 服务不可用、延迟飙升、报错率升高、资源打满
- 值班同学需要先做止血，再进入深入排查

症状：
- 用户反馈无法访问
- 监控平台出现红线或大面积告警
- 接口超时、5xx 增长、队列积压、Pod/进程反复重启

快速判断：
- 问题是否仍在持续
- 影响范围是单实例、单机房、单租户，还是全局
- 最近 30 分钟是否有发布、扩容、配置变更、证书更新、依赖切换
- 是否有明确止血动作可以先执行

修复步骤：
1. 先确认影响范围
- 记录：
  - 开始时间
  - 影响系统
  - 影响用户范围
  - 当前症状
- 快速查看：

```bash
curl -I https://service.example.com
systemctl status service-name --no-pager
kubectl get pods -A
```

2. 判断是否需要立即止血
- 常见止血动作：
  - 回滚最近版本
  - 下线异常实例
  - 重启卡死进程
  - 扩容副本
  - 暂时关闭高风险新功能
- 如果止血动作存在数据风险，先明确风险再执行

3. 检查最近变更
- 优先核对：
  - 发布记录
  - 配置变更
  - 证书变更
  - 定时任务变更
  - 网络策略 / 安全组 / 防火墙变更

4. 检查四类基础资源
- CPU

```bash
top
```

- 内存

```bash
free -h
```

- 磁盘

```bash
df -h
df -i
```

- 网络 / 端口

```bash
ss -ltnp
curl -v http://127.0.0.1:port/health
```

5. 检查日志与健康探针

```bash
journalctl -u service-name -n 200 --no-pager
kubectl logs deployment/name --tail=200
curl http://127.0.0.1:port/health
```

6. 快速分类问题类型
- 资源型：CPU、内存、磁盘、句柄耗尽
- 依赖型：数据库、Redis、MQ、第三方 API 不可用
- 发布型：新版本、新配置、新镜像引入问题
- 网络型：DNS、LB、Ingress、证书、代理
- 数据型：锁等待、慢查询、积压、数据不一致

7. 固定输出状态同步
- 每次同步至少包含：
  - 当前影响范围
  - 已执行动作
  - 当前判断
  - 下一步动作
  - 风险或回滚点

回滚：
- 如果事故发生在变更后且已有明确回滚路径，优先回滚到最近稳定版本
- 回滚前确认：
  - 数据兼容性
  - 配置兼容性
  - 数据库迁移是否可逆

验证：
- 核心接口恢复成功率
- 健康检查恢复
- 关键业务链路抽样成功
- 告警恢复或明显下降
- 回滚/止血后 10 到 15 分钟内无再次恶化

风险提示：
- 没有确认影响范围前，不要同时做多个高风险动作
- 不要在事故进行中做“顺手升级”
- 没有日志和指标证据时，不要过早下结论
- 对数据库和消息队列的修复动作要先评估数据一致性风险
