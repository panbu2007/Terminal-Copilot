# 工作区与配置备份

适用场景：
- 需要在发布、升级或高风险操作前备份项目目录与配置
- 需要为 AI 工具、服务配置、runbook 数据、secrets 建立回滚点
- 需要把“代码备份”和“配置备份”分开治理

症状：
- 发布后想回滚，但没有当前配置快照
- 代码可回退，环境变量和本地配置却不可回退
- secrets、systemd 配置、反代配置散落多处

快速判断：
- 当前有哪些目录和配置真正需要备份
- 哪些由 Git 管理，哪些不由 Git 管理

修复步骤：

1. 列出备份范围
- 代码目录
- `.env` / secrets 目录
- systemd unit / drop-in
- Nginx 配置
- Docker Compose 文件
- 数据卷和数据库备份点

2. 建立最小备份清单
- 代码：Git 提交或归档
- 配置：单独导出
- 数据：单独快照或备份

3. 备份前标记版本
- 记录时间、提交号、变更单号、操作者

4. 恢复演练
- 至少在测试环境演练一次“从备份恢复服务”

回滚：
- 恢复代码版本
- 恢复配置文件
- 恢复数据快照
- 按顺序重启依赖服务

验证：
- 备份文件可读取
- 恢复步骤文档齐全
- 测试环境恢复成功

风险提示：
- 只有代码备份、没有配置和数据备份，回滚是不完整的
- secrets 备份必须限制访问权限
- 未演练的备份不能视为可用

关键词：
- backup workspace
- config backup
- rollback point
- service backup
- deploy backup
- 配置备份
- 工作区备份
- 回滚准备
