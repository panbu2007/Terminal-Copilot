# API Key 存储与轮换

适用场景：
- 需要统一存放和轮换模型、数据库、Webhook、SaaS API Key
- 需要规范服务密钥与个人密钥的使用方式
- 怀疑 key 泄露，必须快速处理

症状：
- key 写死在脚本或仓库里
- 同一个 key 被多个系统复用
- 无法确认 key 当前是否仍然有效
- key 被发送到聊天、日志、工单或截图中

快速判断：
- 检查代码仓库、配置文件、shell 历史、CI 配置
- 检查 systemd / Docker / `.env` / secrets 目录

修复步骤：

1. 先收敛存放位置
- 优先存放在：
  - 环境变量
  - secrets 管理系统
  - 单独权限控制的本地 secrets 目录
- 不要存放在：
  - Git 仓库
  - 普通文档
  - 聊天记录
  - 可共享截图

2. 区分 key 类型
- 个人开发 key
- 服务 key
- 生产 key
- 测试 key

3. 轮换步骤
- 新建新 key
- 更新服务配置
- 重启或 reload 应用
- 验证新 key 生效
- 删除旧 key

4. 泄露后的动作
- 立即轮换，不要只做“删除展示位置”
- 检查调用日志和权限范围
- 如支持，限制来源 IP、作用域和过期时间

回滚：
- 保留旧 key 的短时窗口
- 如果新 key 导致业务异常，恢复旧配置并尽快修复

验证：
- 新 key 可正常调用
- 旧 key 已不可用
- 代码仓库和明文配置中不再出现 key

风险提示：
- 聊天消息里出现过的 key 默认按已泄露处理
- 测试环境和生产环境必须分离
- 服务 key 不要和个人账号绑定

关键词：
- api key rotation
- api key storage
- secret leak
- token rotate
- key 泄露
- 密钥存储
- 密钥轮换
- 凭据治理
